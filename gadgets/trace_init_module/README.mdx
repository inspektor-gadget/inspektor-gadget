# trace_init_module

The trace_init_module gadget emits events when processes invoke the `init_module()` or `finit_module()` syscalls to load kernel modules.

**Syscalls traced:**
- `init_module(module_image, len, param_values)` - loads a module from a memory buffer
- `finit_module(fd, param_values, flags)` - loads a module from a file descriptor

This gadget captures:
- For `init_module`: module size (`len`) and parameters (`param_values`)
- For `finit_module`: file descriptor (`fd`), resolved filepath, flags, and parameters (`param_values`)

**Note:** The `module_image` parameter from `init_module` is intentionally not captured.

## Requirements

- Minimum Kernel Version : 5.4

## Flags

### `--pid`

Show only events generated by processes with this pid

Default value: ""

### `--uid`

Show only events generated by processes with this uid

Default value: ""

## Guide

This example shows how to observe a kernel module being loaded on a test node.
Because kernel modules affect the host, use a disposable node and make sure the
`dummy` module is not already in use before following the example.

First, run the gadget on the node where you will load the module. Replace
`worker-1` with the name of your test node:

<Tabs groupId="env">
  <TabItem value="kubectl-gadget" label="kubectl gadget">
```bash
$ kubectl gadget run trace_init_module:%IG_TAG% --node worker-1 --fields=comm,pid,syscall,len,filepath,flags,param_values
```
  </TabItem>

  <TabItem value="ig" label="ig">
```bash
$ sudo ig run trace_init_module:%IG_TAG% --fields=comm,pid,syscall,len,filepath,flags,param_values
```
  </TabItem>
</Tabs>

In another terminal on that node, load the `dummy` network module with one
interface:

```bash
$ sudo modprobe dummy numdummies=1
```

The gadget reports the `finit_module` syscall used by `modprobe`:

```bash
COMM     PID   SYSCALL      LEN FILEPATH                                      FLAGS PARAM_VALUES
modprobe 14527 finit_module   0 /lib/modules/6.8.0/kernel/drivers/net/dummy.ko     0 numdummies=1
```

The `filepath` field identifies the module file, while `param_values` shows the
options passed to it. The `len` field is zero because it only applies to the
`init_module` syscall; `finit_module` loads the module from the file descriptor
represented by `filepath`.

Stop the gadget with Ctrl-C, then remove the test module:

```bash
$ sudo modprobe -r dummy
```
