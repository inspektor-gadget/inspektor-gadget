# trace_exec

The trace_exec gadget notifies when new processes are executed.

## Description 

This gadget tracks all executed binaries on a node or within containers. It helps monitor process execution events in real-time, providing details like process name, arguments, and whether the binary was modified (using the Upper_Layer field).

## Use Cases

### Security Monitoring
- Detect unfamiliar or unauthorized binaries.
- Flags modified programs via the Upper_Layer field (e.g.. injected malware).

### Threat Detection
- Spot unexpected or suspicious execution patterns inside containers.

### Compliance Auditing
- Maintain execution logs for auditing;
- Verify binaries are from trusted container layer

### DevSecOps
- Detect accidental debug tools (e.g., gdb, strace) in production and enforce runtime policies.

### Debugging & Troubleshooting
- Debug application errors caused by missing or restricted files.
- Investigate unexpected file access patterns.

## Requirements

- Minimum Kernel Version : 5.4

## Flags

### `--ignore-failed`

Ignore failed events

Default value: "true"

### `--paths`

Show current working directory, file path and executable path. The file path and executable path might be different in case of a script.

Default value: "false"

### `--uid`

Show only events generated by processes with this uid

Default value: ""

## Limitations

- The upper layer field is only available when the executable is executed
 correctly (ret=0). For example, if the executable does not have the "execute"
 permission, the execution will fail and the upper layer field will not be
 defined.
- The file path might not be set when the execution fails (ret!=0), depending
 on the error.
- The exepath is set to the current process and not the new process when the
 execution fails (ret!=0).
